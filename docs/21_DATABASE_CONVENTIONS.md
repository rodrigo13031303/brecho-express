# Convenções de Banco de Dados - Brechó Express

## 1. Objetivo do documento

Este documento define as convenções oficiais de banco de dados do Brechó Express. Ele serve como referência obrigatória para a criação de DBML, Oracle SQL, packages, constraints, índices, triggers, views, jobs e futuras migrations.

O propósito principal é garantir consistência técnica, alinhamento com o domínio e padronização entre negócio, banco de dados, APIs e aplicação Flutter.

## 2. Princípios gerais

- Poucas convenções, seguidas por todos.
- O domínio vem antes do banco.
- Toda tabela deve existir no Data Dictionary antes do Oracle.
- Nenhum SQL deve ser criado sem documentação prévia.
- A nomenclatura técnica deve seguir o domínio oficial.

## 3. Convenção de tabelas

- Todas as tabelas usam prefixo BEX_.
- Tabelas usam nome completo da entidade.
- Exemplos corretos:
  - BEX_PROFILE
  - BEX_ACCOUNT
  - BEX_STORE
  - BEX_PRODUCT
  - BEX_PURCHASE_REQUEST
- Exemplos proibidos:
  - BEX_PFL
  - BEX_ACC
  - BEX_STR
  - BEX_PRD
  - BEX_PUR

## 4. Convenção de siglas

- Cada entidade possui sigla única de 3 letras.
- A sigla é usada em colunas, PKs, FKs, índices e packages.
- Exemplos:
  - PROFILE = PFL
  - ACCOUNT = ACC
  - STORE = STR
  - PRODUCT = PRD
  - ORDER = ORD
  - PAYMENT = PYM
  - SHIPMENT = SHP

## 5. Convenção de colunas

- Colunas próprias usam prefixo da entidade.
- Exemplo:
  - PFL_ID
  - PFL_PUBLIC_ID
  - PFL_NAME
  - PFL_STATUS
- FKs usam a sigla da entidade referenciada.
- Exemplo:
  - STR_ID em BEX_PRODUCT referencia BEX_STORE
  - PFL_ID em BEX_STORE_USER referencia BEX_PROFILE

## 6. Chaves primárias

- Toda tabela terá ID numérico interno.
- Tipo padrão:
  NUMBER(19) GENERATED BY DEFAULT AS IDENTITY
- A PK segue o padrão:
  <SIGLA>_ID
- Exemplo:
  PFL_ID, ACC_ID, STR_ID, PRD_ID

## 7. Identificador público

- Toda entidade exposta pela API terá PUBLIC_ID.
- Padrão:
  <SIGLA>_PUBLIC_ID
- Tipo:
  VARCHAR2(32)
- Deve ser único.
- APIs externas devem usar PUBLIC_ID, nunca o ID interno.
- O ID interno é exclusivo do Oracle.

## 8. Auditoria

- Toda tabela principal deve possuir:
  <SIGLA>_CREATED_AT
  <SIGLA>_UPDATED_AT
  <SIGLA>_CREATED_BY
  <SIGLA>_UPDATED_BY
- CREATED_BY e UPDATED_BY referenciam BEX_PROFILE.<PFL_ID>.
- Para operações automáticas, será usado um Profile técnico SYSTEM.

## 9. Status e soft delete

- Não usar DELETE físico para entidades de negócio.
- Exclusão lógica deve ser feita por STATUS.
- Padrões sugeridos:
  ACTIVE
  INACTIVE
  ARCHIVED
  BLOCKED
- Tabelas transacionais podem possuir status próprios.

## 10. Tipos de dados padrão

- IDs: NUMBER(19)
- PUBLIC_ID: VARCHAR2(32)
- Textos curtos: VARCHAR2
- Descrições longas: CLOB
- Valores monetários: NUMBER(12,2)
- Quantidades: NUMBER(12,3) ou NUMBER(12,0), conforme domínio
- Datas: TIMESTAMP
- Flags: CHAR(1), usando S/N ou Y/N conforme definição futura

## 11. Constraints

- PK: PK_<SIGLA>
- FK: FK_<SIGLA_ATUAL>_<SIGLA_REFERENCIADA>
- UK: UK_<SIGLA>_<CAMPO>
- CK: CK_<SIGLA>_<REGRA>
- Exemplos:
  PK_PFL
  FK_STR_PFL
  UK_PFL_PUBLIC_ID
  CK_PFL_STATUS

## 12. Índices

- Índices devem seguir:
  IDX_<SIGLA>_<DESCRICAO>
- Exemplos:
  IDX_PFL_PHONE
  IDX_PRD_STATUS
  IDX_STR_LOCATION
- Índices devem existir por necessidade real de consulta.

## 13. Packages Oracle

- Cada domínio terá packages próprias.
- Padrão:
  <SIGLA>_API_PKG
  <SIGLA>_RULE_PKG
- API_PKG expõe operações consumidas pelo ORDS.
- RULE_PKG concentra regras de negócio.
- Exemplos:
  PFL_API_PKG
  PFL_RULE_PKG
  PRD_API_PKG
  PRD_RULE_PKG

## 14. ORDS e APIs

- APIs REST devem usar nomes de domínio.
- APIs externas usam PUBLIC_ID.
- O Flutter nunca acessa IDs internos.
- O Flutter nunca conhece SQL.
- ORDS chama packages Oracle, não SQL solto.

## 15. DBML

- O DBML deve seguir as mesmas convenções.
- O DBML nasce depois do Data Dictionary.
- O Oracle SQL nasce depois do DBML.

## 16. Classificação de entidades

As entidades do domínio devem ser classificadas conforme o seu papel no sistema:

- MASTER
- TRANSACTION
- SUPPORT
- CONFIGURATION
- AUDIT

## 17. Regras proibidas

- Não criar tabela sem Data Dictionary.
- Não criar SQL antes do DBML.
- Não usar tabela abreviada.
- Não expor ID interno na API.
- Não usar DELETE físico para entidade de negócio.
- Não criar package genérica tipo PKG_UTIL.
- Não colocar regra de negócio no Flutter.

## 18. Conclusão

Este documento é a Constituição do Banco do Brechó Express. Ele estabelece as regras mínimas, claras e obrigatórias para garantir que o modelo de dados, a implementação Oracle, as APIs e o aplicativo Flutter evoluam de forma consistente, alinhada ao domínio e à arquitetura do projeto.
