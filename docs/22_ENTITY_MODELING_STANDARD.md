# Padrão Oficial de Modelagem de Entidades - Brechó Express

## 1. Objetivo

Este documento define o padrão oficial de modelagem de entidades do Brechó Express. Ele deve ser utilizado como referência obrigatória por qualquer IA, analista ou desenvolvedor na criação de novas entidades do Data Dictionary.

O objetivo principal é garantir consistência entre documentação, banco de dados, APIs e aplicação Flutter, preservando a arquitetura, a Linguagem Ubíqua e as decisões de negócio já consolidadas.

## 2. Estrutura obrigatória

Toda entidade do projeto deve conter obrigatoriamente as seguintes seções:

- Ficha Técnica
- Objetivo
- Classificação
- Responsabilidades
- Não é responsabilidade
- Dono da Informação
- Regras de Negócio
- Relacionamentos
- Atributos
- Índices
- Packages Oracle
- APIs
- Flutter
- Observações

## 3. Convenções Gerais

As entidades devem seguir as convenções abaixo:

- PK padrão:
  NUMBER(19) GENERATED BY DEFAULT AS IDENTITY

- PUBLIC_ID:
  CHAR(32)

- Soft Delete obrigatório

- Auditoria obrigatória

- Campos padrão:
  - STATUS
  - CREATED_AT
  - UPDATED_AT
  - CREATED_BY
  - UPDATED_BY

## 3.1 Classificação das Entidades

As entidades do Brechó Express devem ser classificadas conforme seu papel no domínio.

### MASTER

Entidades centrais do domínio, que representam conceitos principais e possuem identidade própria.

Exemplos:
- PROFILE
- ACCOUNT
- STORE
- PRODUCT
- ORDER

### SUPPORT

Entidades auxiliares ou associativas que dão suporte a uma entidade principal ou a um relacionamento.

Exemplos:
- ADDRESS
- PROFILE_ROLE
- STORE_USER
- PRODUCT_IMAGE

### CONFIGURATION

Entidades parametrizáveis ou de configuração do sistema.

Exemplos:
- ROLE
- CATEGORY
- BRAND
- STORE_PLAN
- PRODUCT_STATUS

### TRANSACTION

Entidades que representam operações, fluxos ou eventos transacionais do negócio.

Exemplos:
- CART
- PURCHASE_REQUEST
- ORDER
- PAYMENT
- SHIPMENT

### AUDIT

Entidades voltadas para histórico, rastreabilidade, logs ou auditoria.

Exemplos:
- ACTIVITY_LOG
- AUDIT_LOG
- LOGIN_HISTORY

## 4. Convenção de Auditoria

Toda entidade deve possuir:

- <SIGLA>_CREATED_AT
- <SIGLA>_UPDATED_AT
- <SIGLA>_CREATED_BY
- <SIGLA>_UPDATED_BY

CREATED_BY e UPDATED_BY sempre referenciam PROFILE.

Operações automáticas utilizam o PROFILE técnico SYSTEM.

## 5. Convenção de Relacionamentos

Os relacionamentos devem ser documentados utilizando:

- 1:1
- 1:N
- N:1
- N:N através de entidade associativa

## 6. Convenção de Regras de Negócio

Todas as regras de negócio devem utilizar a numeração:

- RN-001
- RN-002
- RN-003
- ...

As regras devem descrever comportamento de negócio, nunca implementação técnica.

## 7. Convenção de Responsabilidades

A seção Responsabilidades deve responder:

- O que esta entidade faz?

A seção Não é responsabilidade deve responder:

- O que esta entidade nunca deve fazer?

## 8. Convenção de Índices

Toda entidade deve possuir:

- PK
- UK_PUBLIC_ID

Índices adicionais devem existir apenas quando houver necessidade funcional ou de consulta.

Evitar criar índices desnecessários.

## 9. Convenção de Packages

Toda entidade deverá possuir:

- <SIGLA>_API_PKG
- <SIGLA>_RULE_PKG

API_PKG expõe operações consumidas pelo ORDS.

RULE_PKG concentra regras de negócio.

## 10. Convenção de APIs

As APIs devem:

- Sempre utilizar PUBLIC_ID.
- Nunca expor ID interno.
- Seguir o padrão REST.

## 11. Convenção Flutter

Toda entidade deve documentar:

- Model
- Repository
- Controller
- Page

O Flutter nunca deve acessar SQL diretamente.

## 12. Convenção de Observações

A seção Observações deve ser utilizada para registrar:

- decisões arquiteturais;
- reutilização futura;
- restrições importantes;
- contexto de negócio.

## 13. Checklist de Revisão

Antes de considerar uma entidade concluída, verificar:

- Segue a Linguagem Ubíqua.
- Respeita o Data Dictionary.
- Utiliza PUBLIC_ID CHAR(32).
- Possui auditoria.
- Possui Soft Delete.
- Possui STATUS.
- Possui responsabilidades claras.
- Possui "Não é responsabilidade".
- APIs usam PUBLIC_ID.
- Flutter não conhece ID interno.
- Está alinhada às Convenções do Banco.
- Está alinhada às ADRs do projeto.
